# **SES事業における人材開発と評価制度の構築：未経験採用からスペシャリスト・経営層（1〜20年目）に至る包括的成長ロードマップ**

## **第1章 戦略的背景とSES評価制度の構造的課題**

システムエンジニアリングサービス（SES）業界において、未経験者を大量に採用し育成するビジネスモデルは、日本のIT人材不足（2030年問題）に対する有効な解であると同時に、極めて難易度の高い組織課題を内包しています。SESの構造上、社員は「自社」に雇用されながら「顧客企業」の現場で稼働するため、帰属意識の希薄化、評価の不透明性、そしてキャリアパスの断絶が起きやすく、これが高い離職率に直結します1。

本レポートでは、未経験者が業界標準レベルのエンジニアへと成長し、さらには高単価なスペシャリストやマネジメント層へと進化するための20年間のロードマップを提示します。また、その成長を定量・定性の両面から測定するための「5つのコンピテンシー（能力特性）」と「技術領域別スキルマトリクス」を定義します。これらは、単なる査定ツールではなく、エンジニア自身が市場価値（単価）を認識し、自律的なキャリア形成を促すための羅針盤として機能するよう設計されています。

### **1.1 SES特有の「評価の二重性」と解消アプローチ**

SESにおける評価の最大の課題は、評価者（自社マネージャー）と被評価者（現場エンジニア）が日常的に接点を持たないという「情報の非対称性」にあります。

1. **顧客評価（現場評価）**: 成果物の品質、納期遵守、現場でのコミュニケーション能力。これは契約継続や単価交渉に直結します3。
2. **自社評価（組織貢献）**: 帰社日への参加、後輩のメンター業務、自社の技術力底上げへの寄与など。これは組織の求心力維持に不可欠です5。

本レポートで提案する評価制度は、この二重性を統合し、半期ごとのMBO（目標管理制度）を通じて「現場でのパフォーマンス向上」が「自社の利益」と「個人の給与アップ」に連動するメカニズムを構築することを目的とします6。

## **第2章 評価カテゴリの妥当性検証と定義**

ご提示いただいた5つの評価カテゴリ（マネジメント、リーダーシップ、コミュニケーション、技術、ビジネススキル）は、SESエンジニアのキャリア形成において極めて高い妥当性を持ちます。多くのSES企業が技術偏重の評価に陥る中、ビジネススキルやヒューマンスキルを明示的に評価項目化することは、エンジニアの「単価（市場価値）」を最大化するために不可欠です。以下に各カテゴリの定義と、SES事業における必然性を論じます。

### **2.1 マネジメント（Management）**

**定義**: 目標達成のために、リソース（人、モノ、金、時間）を効率的に管理・運用する能力8。 **SESにおける妥当性**: 一般に管理職のスキルと誤解されがちですが、SESでは新人段階から「自己管理（Self-Management）」として求められます。勤怠管理、健康管理、タスクの納期管理は、SES契約の根幹である「善管注意義務（準委任契約）」を履行するための最低条件です9。ベテラン層においては、プロジェクトの進捗管理（QCD管理）やチームの収益管理へと発展します。

* **検証**: 納期遅延や勤怠不良は契約解除（現場退場）の直接的原因となるため、このカテゴリは必須です。

### **2.2 リーダーシップ（Leadership）**

**定義**: ビジョンや方向性を示し、周囲（チームメンバーや顧客）を動機づけ、変革や課題解決を主導する能力10。 **SESにおける妥当性**: マネジメントが「守り（管理）」であるのに対し、リーダーシップは「攻め（影響力）」です。常駐先において、自社の若手メンバーをまとめ上げる現場リーダーとしての役割や、顧客の潜在的な課題を発見し解決策を提案する姿勢は、リーダーシップ評価となります8。

* **検証**: 単価向上（増員提案やリーダー単価へのシフト）には、技術力以上に現場での影響力が関与するため、キャリアアップの要件として妥当です。

### **2.3 コミュニケーション（Communication）**

**定義**: 情報を正確に伝達し、相手の意図を正確に汲み取り、合意形成を図る能力9。 **SESにおける妥当性**: SESエンジニアにとって、コミュニケーション能力は「生存スキル」です。未知の環境（常駐先）に適応し、顧客の要望（要件）をヒアリングし、トラブル発生時に迅速に報告・連絡・相談（ホウレンソウ）を行う能力がなければ、いかに高い技術があっても信頼を得られません11。

* **検証**: クレームの多くは技術的なバグではなく、報告の遅れや認識の齟齬から発生します。よって、最重要の評価項目の一つです。

### **2.4 技術スキル（Technical Skills）**

**定義**: 業務遂行に必要な専門的知識と技術力。プログラミング、設計、ツール操作など3。 **SESにおける妥当性**: エンジニアとしての商品価値そのものです。ただし、SESでは「広く浅い知識」よりも、案件需要にマッチした「特定の深掘り」あるいは「フルスタック性」が評価されます。本レポートでは、後述の通りフロントエンド、バックエンド、インフラ、クラウドに細分化して評価します。

* **検証**: 未経験からスタートする場合、技術習得のロードマップが明確でなければ成長が止まります。よって、最も詳細な粒度が求められるカテゴリです。

### **2.5 ビジネススキル（Business Skills）**

**定義**: コスト意識、契約知識、業界知識、折衝力など、ビジネスパーソンとしての基礎能力9。 **SESにおける妥当性**: SESエンジニアは、現場における「営業の最前線」にいます。自社の売上構造（単価×稼働率）、契約形態（派遣・準委任・請負）の違い、顧客の業界動向を理解することは、自社の利益貢献に直結します14。

* **検証**: 「技術だけやっていればいい」という思考は、30代以降のキャリア停滞（単価の頭打ち）を招きます。エンジニアに経営視点を持たせるために不可欠なカテゴリです。

**第3章 20年成長ロードマップ：フェーズ別期待役割と到達目標**

エンジニアの成長プロセスを、未経験入社から20年目まで、5つのフェーズに分割します。各フェーズにおいて、前述の5カテゴリで求められる水準と、想定される市場価値（単価レンジ）を定義します。

### **フェーズ1：立ち上がり・基盤形成期（1〜3年目）**

**テーマ**: 「プロフェッショナルとしての自立と信頼獲得」

**想定単価レンジ**: 40万〜60万円/月

**重点比重**: 技術(50%)、コミュニケーション(30%)、マネジメント(20%)

この時期は、顧客にとって「手のかからない要員」になることが最優先です。未経験者は初期において、技術力不足により生産性がマイナスになることもあります。それを補うのが、「報告の正確さ」「勤勉さ」「学習意欲」です16。

| カテゴリ | 具体的な到達目標（行動指標） |
| :---- | :---- |
| **技術** | 指示された仕様通りにコードを書ける（Java/PHP等）。単体テストを完遂できる。基本情報技術者レベルのIT基礎知識17。 |
| **コミュ** | 「ホウレンソウ」の徹底。分からないことを放置せず、適切なタイミングで質問できる。遅刻・欠勤の連絡をルール通り行える11。 |
| **マネジメント** | 自己のタスク管理。納期（期限）を守る意識。体調管理と勤怠の安定9。 |
| **リーダー** | チームのルールを遵守する。後輩（2年目以降）が入ってきた際に、基本ルールを教える。 |
| **ビジネス** | SESの商流や契約の仕組みを理解する。セキュリティ事故（情報漏洩）の怖さを理解し、遵守する5。 |

### **フェーズ2：中堅・主力プレイヤー期（4〜7年目）**

**テーマ**: 「チームの中核としての成果創出」

**想定単価レンジ**: 60万〜80万円/月

**重点比重**: 技術(40%)、コミュ(30%)、リーダーシップ(20%)、ビジネス(10%)

現場の業務知識（ドメイン知識）が蓄積され、詳細設計から実装、テストまでを一人称で完遂できるレベルです。顧客からの信頼も厚くなり、指名で案件継続を依頼されるようになります4。

| カテゴリ | 具体的な到達目標（行動指標） |
| :---- | :---- |
| **技術** | 基本設計書の作成。フレームワークの特性を理解した実装。DB設計（正規化等）の理解。トラブルシューティングの自律的解決19。 |
| **コミュ** | 顧客担当者との仕様調整。専門用語を使わず、わかりやすく説明する能力。ドキュメント（設計書・マニュアル）の品質向上9。 |
| **マネジメント** | サブチーム（2〜3名）の進捗管理。工数見積もりの精度向上（予実差±20%以内）20。 |
| **リーダー** | 新人・後輩のOJT担当。コードレビューの実施。ポジティブな態度の波及9。 |
| **ビジネス** | 顧客のビジネス課題への関心。現場での増員ニーズや別案件情報のキャッチアップと営業への連携（現場営業）14。 |

### **フェーズ3：リーダー・スペシャリスト分岐期（8〜12年目）**

**テーマ**: 「組織への影響力拡大と高付加価値化」

**想定単価レンジ**: 80万〜100万円/月

**重点比重**: バランス型（PM志向） または 技術特化型（スペシャリスト志向）

キャリアが「プロジェクトマネジメント（PM/PL）」と「高度技術スペシャリスト（Tech Lead）」に分岐します。どちらに進むにせよ、顧客の潜在ニーズを解決する提案力が求められます21。

| カテゴリ | 到達目標（PMルート） | 到達目標（スペシャリストルート） |
| :---- | :---- | :---- |
| **技術** | 幅広い技術知識（フルスタック性）。アーキテクチャ選定の判断能力。 | 特定領域（クラウド、AI、セキュリティ等）における深い専門性。難易度の高いバグ解析やパフォーマンスチューニング22。 |
| **コミュ** | 顧客折衝、要件定義のファシリテーション。利害関係者間の調整12。 | 技術的な正当性の論理的説明。技術顧問的な助言。 |
| **マネジメント** | プロジェクト全体のQCD管理。リスク管理。要員計画の策定8。 | 技術的負債の管理。開発プロセスの効率化（CI/CD導入等）。 |
| **リーダー** | チームビルディング。メンバーのメンタルケアとモチベーション管理10。 | 技術選定のリード。勉強会主催などによる技術力底上げ9。 |
| **ビジネス** | 見積書作成支援。契約更新交渉の支援。顧客満足度の最大化23。 | 技術ブランディングによる自社価値向上。 |

### **フェーズ4：マネージャー・アーキテクト期（13〜16年目）**

**テーマ**: 「事業戦略との連動と組織構築」

**想定単価レンジ**: 100万〜130万円/月

**重点比重**: ビジネス(40%)、マネジメント(30%)、技術(30%)

単一のプロジェクトを超え、複数プロジェクトの統括や、全社的な技術標準の策定に関わります。SES企業においては、現場に出ながらも自社の課長・部長職を兼務するケースが多いフェーズです。

* **技術**: 全社的な技術戦略の立案。新技術の導入検証。技術教育カリキュラムの作成。
* **ビジネス**: アカウントマネジメント（重要顧客の囲い込み）。新規サービスやソリューションの開発提案24。
* **マネジメント**: 組織設計。評価制度の運用。採用面接官としての目利き。

### **フェーズ5：エグゼクティブ・プリンシパル期（17〜20年目以降）**

**テーマ**: 「業界における権威と経営参画」

**想定単価レンジ**: 130万円〜/月、または役員報酬

* **役割**: CTO（最高技術責任者）、VPoE（技術組織責任者）、または事業部長。
* **期待値**: 経営課題を技術で解決する。業界内での登壇や執筆を通じたプレゼンス向上。次世代リーダーの育成21。

**第4章 技術スキルの詳細細分化と評価基準**

ご要望に基づき、技術スキルを「フロントエンド」「バックエンド」「インフラ」「クラウド」の4領域に細分化し、それぞれの評価基準（スキルマップ）を策定します。SESでは案件ごとに使用技術が異なるため、汎用的な基礎力と、トレンド技術への適応力の両面を評価します。

### **4.1 フロントエンド（Frontend）領域**

**特性**: 技術の移り変わりが激しく、デザイン（UI/UX）への理解も求められる領域です17。

| レベル | 評価基準・スキルセット例 |
| :---- | :---- |
| **Lv.1 (初級)** | HTML5/CSS3の正確なコーディング。レスポンシブデザインの実装。JavaScript(ES6)の基本構文理解。jQueryの読解・修正。Gitの基本操作（commit, push, pull）17。 |
| **Lv.2 (中級)** | モダンフレームワーク（React, Vue.js, Angular）を用いたコンポーネント設計・実装。TypeScriptによる型定義。非同期通信（API連携）の実装。CSSプリプロセッサ（Sass/SCSS）の活用17。 |
| **Lv.3 (上級)** | フロントエンドのパフォーマンス最適化（レンダリング速度改善）。状態管理ライブラリ（Redux, Vuex等）の適切な設計。Next.js/Nuxt.jsを用いたSSR/SSGの実装。テストコード（Jest, Cypress）の作成17。 |
| **Lv.4 (特級)** | マイクロフロントエンドアーキテクチャの設計。BFF（Backend For Frontend）の設計・構築。WebAssemblyの活用。UI/UX改善提案によるCVR向上への貢献26。 |

### **4.2 バックエンド（Backend）領域**

**特性**: システムの安定性、データ整合性、セキュリティが重視されます。SES案件数が最も多く、単価も安定しています19。

| レベル | 評価基準・スキルセット例 |
| :---- | :---- |
| **Lv.1 (初級)** | 1つの言語（Java, PHP, Python, Ruby等）での基本構文理解。MVCフレームワーク（Spring, Laravel等）を用いたCRUD機能の実装。SQLの基本的な操作（SELECT, INSERT, UPDATE, JOIN）17。 |
| **Lv.2 (中級)** | データベース設計（ER図作成、正規化）。RESTful APIの設計・実装。セキュリティ対策（SQLインジェクション、XSS対策）。単体・結合テストの実施。Linuxコマンドによる基本操作19。 |
| **Lv.3 (上級)** | パフォーマンスチューニング（インデックス最適化、クエリ改善）。非同期処理・バッチ処理の設計。Dockerコンテナを活用した開発環境構築。CI/CDパイプラインの構築・運用18。 |
| **Lv.4 (特級)** | マイクロサービスアーキテクチャの設計。DDD（ドメイン駆動設計）の実践。大規模トラフィック対策（キャッシュ戦略、負荷分散）。NoSQL（DynamoDB等）の選定・活用19。 |

### **4.3 インフラ（Infrastructure）領域**

**特性**: オンプレミスからクラウドへの移行期ですが、ネットワークやOSの基礎知識は依然として不可欠です28。

| レベル | 評価基準・スキルセット例 |
| :---- | :---- |
| **Lv.1 (初級)** | 手順書に基づくサーバー構築・設定変更。監視ツール（Zabbix等）のアラート対応。Linux/Windows Serverの基本コマンド操作。ネットワーク基礎（TCP/IP, DNS, DHCP）の理解29。 |
| **Lv.2 (中級)** | サーバーの設計・構築（Web/AP/DB構成）。ネットワーク機器（ルータ, スイッチ, FW）の設定。シェルスクリプト（Bash, PowerShell）による作業自動化。仮想化技術（VMware, Hyper-V）の操作25。 |
| **Lv.3 (上級)** | 可用性設計（冗長化、クラスタリング）。バックアップ・DR（災害復旧）設計。セキュリティポリシーの策定と実装（WAF, IPS/IDS）。構成管理ツール（Ansible）の活用29。 |
| **Lv.4 (特級)** | ハイブリッドクラウド環境の設計。大規模ネットワークのトラフィック制御。SRE（Site Reliability Engineering）の実践による運用自動化・信頼性向上18。 |

### **4.4 クラウド（Cloud）領域**

**特性**: AWS, Azure, Google Cloudなど、プラットフォーム固有の知識に加え、「Infrastructure as Code (IaC)」のスキルが高単価の鍵となります28。

| レベル | 評価基準・スキルセット例 |
| :---- | :---- |
| **Lv.1 (初級)** | クラウドコンソール（GUI）を用いた基本リソース（EC2, S3, RDS等）の作成・削除。IAMによるユーザー管理。クラウドの基本概念とコスト構造の理解。AWS CLF/SAA等の資格取得28。 |
| **Lv.2 (中級)** | VPC設計（サブネット、ルートテーブル、セキュリティグループ）。ロードバランサー（ELB）の設定。CloudWatch等を用いた監視設定。マネージドサービス（RDS, Lambda）の活用28。 |
| **Lv.3 (上級)** | IaCツール（Terraform, CloudFormation）を用いたインフラコード化。コンテナオーケストレーション（EKS, ECS, Kubernetes）の設計・運用。サーバーレスアーキテクチャの構築18。 |
| **Lv.4 (特級)** | マルチクラウド・マルチリージョン設計。FinOps（クラウドコスト最適化）の実践。クラウドネイティブセキュリティの設計（GuardDuty, Security Hub）。DevSecOpsの推進19。 |

**第5章 半期ごとのMBO（目標管理）評価シートの運用設計**

評価制度を形骸化させないためには、上記ロードマップを「半期ごとの具体的アクション」に落とし込むMBOシートの設計が重要です。ここでは、MBOシートの構造と具体的な記入例を提示します。

### **5.1 評価シートの構成要素（テンプレート案）**

評価シートはExcelまたはスプレッドシートで管理し、以下のセクションで構成します31。

1. **基本情報**: 氏名、等級（グレード）、現在の単価、所属プロジェクト概要
2. **定量目標（業績・技術）**: ウェイト50〜60%
3. **定性目標（行動・プロセス）**: ウェイト30〜40%
4. **情意評価（勤務態度）**: ウェイト10%（減点方式が一般的）
5. **総合評価**: S〜Dの5段階

### **5.2 目標設定のSMART原則と記入例**

目標は必ず「SMARTの法則（Specific, Measurable, Achievable, Relevant, Time-bound）」に基づいて設定させます9。

**記入例：入社3年目（バックエンドエンジニア）の場合**

| カテゴリ | ウェイト | 目標設定（SMARTな記述） | 達成基準（KPI） |
| :---- | :---- | :---- | :---- |
| **技術** | 30% | 現在のJava案件において、Spring Bootの理解を深め、詳細設計書を一人で完遂できるレベルになる。 | ・詳細設計書のレビュー指摘件数5件以下 ・AWS SAA資格を取得する |
| **ビジネス** | 20% | 現場の増員ニーズを把握し、営業に情報を連携する。または、自身の単価5万円アップの根拠となる実績を作る。 | ・増員情報の連携1件以上 ・顧客からの「指名」による契約更新獲得 |
| **コミュ** | 20% | 朝会での進捗報告を1分以内で明瞭に行う。質問時は「現状・調査結果・仮説」をセットにする。 | ・リーダーからのフィードバック評価「B」以上 |
| **マネジメント** | 20% | 工数遅延を発生させない。遅延見込み時は2日前までにアラートを上げる。 | ・納期遵守率100% ・無断残業ゼロ |
| **情意** | 10% | 帰社日（月1回）に必ず参加し、自社の新人と交流する。 | ・帰社日参加率100% |

### **5.3 評価プロセスの運用（1on1とフィードバック）**

SESの評価における最大の課題は、評価者が普段の仕事ぶりを見ていないことです。これを補うために以下のフローを推奨します。

1. **期初面談**: 本人とマネージャー（またはTL）で目標をすり合わせる。会社が期待する役割（ロードマップ上の位置）を明確に伝える。
2. **中間面談（3ヶ月目）**: 進捗確認。現場の状況が変わっていれば目標を修正する。SESでは現場異動が頻繁にあるため、柔軟な変更が必須です。
3. **期末評価**:
   * **自己評価**: 本人が事実に基づいて振り返る29。
   * **顧客フィードバック**: 営業担当が顧客からヒアリングした「顧客満足度」や「クレーム有無」を加味する33。
   * **絶対評価と相対評価の調整**: 単価向上や資格取得などの「絶対的成果」と、全社員の中での「相対的貢献」を調整して決定する。

### **5.4 待機期間（案件・現場がない期間）の評価**

SESでは案件の端境期に「待機」が発生することがあります。この期間の評価は「学習成果」と「社内貢献」に切り替えます34。

* **目標例**: 「待機期間中にポートフォリオサイトを作成し、GitHubに公開する」「社内研修の資料をアップデートする」。これらを達成すれば、待機中でも評価を下げない仕組みにします。

## ---

**第6章 評価と報酬・単価の連動**

評価結果が給与に反映されなければ、制度は機能しません。特にSESでは「単価」という客観的な指標があるため、これと連動させることで透明性を高めることができます。

### **6.1 グレード別単価連動モデル（例）**

評価カテゴリの達成度合いに応じて社内グレードを上げ、給与レンジと期待単価を連動させます。

| 社内グレード | 想定年次 | 期待単価（月） | 年収目安 | 求められる役割 |
| :---- | :---- | :---- | :---- | :---- |
| **G1 (Trainee)** | 1年目 | 〜45万 | 300万〜 | 学習、ルール遵守、勤怠安定 |
| **G2 (Junior)** | 2〜3年目 | 50〜60万 | 350万〜 | 自律的遂行、ホウレンソウ徹底 |
| **G3 (Middle)** | 4〜7年目 | 60〜80万 | 450万〜 | チーム中核、後輩指導、設計能力 |
| **G4 (Senior)** | 8〜12年目 | 80〜100万 | 600万〜 | PL/PM、スペシャリスト、増員貢献 |
| **G5 (Principal)** | 13年目〜 | 100万〜 | 800万〜 | 組織マネジメント、事業企画、技術統括 |

※年収目安は業界水準に基づく参考値16。

### **6.2 「単価還元」の考え方**

近年、SES業界では「単価連動型給与（還元率開示）」がトレンドですが、未経験者を多く採用する場合、初期教育コストがかかるため、完全連動はリスクがあります。

* **推奨モデル**:
  * **低グレード帯（G1-G2）**: 固定給重視。単価が低くても生活給を保証する代わりに、教育を受ける立場であることを明確にする。
  * **高グレード帯（G3以降）**: 成果給比率を高める。単価アップ分の50〜70%を賞与や昇給で還元することで、モチベーション維持と流出防止（独立・転職防止）を図る34。

## ---

**第7章 結論と提言**

未経験者を主力とするSES企業にとって、人材はコストではなく最大の「資産」です。本レポートで提示した「5つのコンピテンシー」と「技術領域別ロードマップ」を導入することで、以下の効果が期待できます。

1. **離職率の低下**: 「この会社にいれば、どのようなエンジニアになれるか」という将来像（20年ロードマップ）が可視化されることで、キャリア不安による離職を防ぎます。
2. **単価の向上**: ビジネススキルやコミュニケーションスキルを評価項目に入れることで、単なる「作業者」ではなく「顧客の課題解決パートナー」としての行動が促され、結果として単価交渉力が向上します。
3. **組織力の強化**: マネジメントやリーダーシップの評価を通じて、現場リーダーが育ち、チーム単位での参画（チーム提案）が可能になります。

### **導入に向けた次のステップ**

* **ステップ1**: 現社員のスキル棚卸しと、暫定グレードの格付け。
* **ステップ2**: MBOシート（Excel版）の配布と、評価者（マネージャー）向けトレーニングの実施。
* **ステップ3**: 半期運用のトライアル実施と、フィードバックによる微修正。

この評価制度は、一度作って終わりではなく、技術トレンドや市場環境の変化に合わせて毎年見直し（アップデート）し続けることが重要です。継続的な改善こそが、変化の激しいIT業界における最強の生存戦略となります。
